# AGENTS.md

## Stack That Matters Here
- Single-package Nuxt 4 app on Bun, not a monorepo. Main source lives under `app/`.
- UI work is built on `@nuxt/ui` 4.x components and dashboard primitives and Tailwind 4. If MCP docs servers for Nuxt/Nuxt UI are available in the session, use them instead of guessing props, slots, or component names.
- Runtime is Bun end-to-end: Nitro uses `preset: 'bun'`, tests run with `bun test`, and production starts with `bun run .output/server/index.mjs`.

## Exact Commands
- Install: `bun install`
- Dev server: `bun run dev` (fixed port `12153`)
- Build: `bun run build`
- Start built app: `bun run start`
- Tests: `bun test` (please note that type errors in app/api-client can be safly ignored because the @hey-api/openapi-ts currently introduces type erros in the generated files)
- Typecheck `bun typecheck`
- Regenerate API client: `bun run api-client:generate`

## Verified Repo Constraints
- There is no checked-in lint or formatter. Do not invent `pnpm`, `npm`, or lint commands that are not defined here.
- `tsconfig.json` references `.nuxt/*`; after dependency or config changes, `nuxt prepare` runs via `postinstall`.
- OpenAPI client code in `app/api-client/` is generated by `@hey-api/openapi-ts`. Do not hand-edit generated `*.gen.ts` files unless the task explicitly targets generated output.
- Client generation pulls from `http://localhost:12151/docs/openapi` and writes to `app/api-client/`. If the backend contract changed, regenerate before debugging frontend type errors.

## App Wiring
- Global app shell is `app/app.vue` -> `UApp` -> `NuxtLayout` -> `NuxtPage`.
- Public pages are mostly SSR, but `nuxt.config.ts` forces `/dashboard/**` and `/auth/**` to `ssr: false`. Avoid relying on server-only behavior inside dashboard/auth pages.
- Auth and role gating happen in global middleware:
  - `app/middleware/auth.global.ts` redirects unauthenticated `/dashboard*` traffic to `/auth/login` using the `leioshub_session_token` cookie.
  - `/dashboard/admin*` access is enforced by checking `user.role === 'admin'`.
  - `app/middleware/rewrites.global.ts` strips trailing slashes.

## Data And API Conventions
- API access is centralized through `app/composables/useAPI.ts`; prefer `useAPI`, `useAPIAsyncData`, and related wrappers over ad hoc fetch logic so auth headers and 401 redirects stay consistent.
- `updateAPIClient()` sets the generated client base URL from runtime config and injects the bearer token. If API calls look unauthenticated, inspect that path first.
- Runtime config keys in use are `LEIOS_API_URL`, `LEIOS_HUB_APP_URL`, and `LEIOS_HUB_APP_ENABLE_SIGNUP`. README names `REPO_API_URL`; trust `nuxt.config.ts` and `example.env` instead.
- Optional dev proxy is controlled by `USE_DEV_PROXY=true` and `DEV_PROXY_TARGET`; it proxies `/api/proxy` through Nitro during development.

## Layout And Styling
- Dashboard navigation and admin/dev section structure are defined in `app/layouts/dashboard.vue`; change that file before scattering nav edits across pages.
- Global styling is intentionally dark-first and minimal in `app/assets/css/main.css` plus `app/app.config.ts`. Preserve the established Nuxt UI + Tailwind approach instead of introducing a parallel styling system.

## Tests
- Current tests are Bun tests under `tests/`; there is no separate frontend test harness configured.
- The existing suite is very small and currently exercises `app/utils/routeMatcher.ts`. For focused verification, run `bun test tests/utiils.test.ts`.
